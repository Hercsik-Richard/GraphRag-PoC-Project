# English

## Overview

This repository is a full-stack proof-of-concept GraphRAG application for source-grounded question answering over uploaded `.txt` documents. The system indexes source text with Microsoft GraphRAG, builds a knowledge graph of extracted entities and relationships, visualizes the graph, and lets the user ask questions through multiple retrieval modes.

The research motivation is explainability and hallucination reduction. Instead of relying only on vector similarity over isolated chunks, the application exposes structured context: source text units, entities, relationships, communities, citations, and query-routing metadata.

This project is topic-independent. It can be used as a document-based knowledge graph assistant for any UTF-8 `.txt` source text.

## Key Features

- `.txt` document upload and background Microsoft GraphRAG indexing.
- Isolated GraphRAG workspaces for each indexed graph under `backend/ragtest/graphs/<graph_id>/`.
- Entity, relationship, community, text-unit, and embedding outputs generated by GraphRAG.
- Interactive React Flow graph visualization with main, connected, and full graph views.
- Chat interface with conversations, message history, source citations, entity citations, and relationship citations.
- Search modes: `auto`, `local`, `global`, `drift`, `source`, and `hybrid`.
- Lightweight auto routing that selects a suitable retrieval mode based on the question.
- Source-grounded answer paths that return `[S1]`, `[S2]` citation metadata with source filename, text-unit id, excerpts, and document ids where available.
- Local Ollama default setup, with optional Gemini and OpenRouter provider support.
- Separate indexing chat, indexing embedding, query chat, and query embedding providers.
- Indexing progress reporting with phase, chunk, percentage, and error state.
- Graph diagnostics for document, text-unit, entity, relationship, community, isolated-node, provider, and embedding-alignment checks.

## Architecture

The application has three main layers:

1. Frontend: upload, provider selection, graph selection, chat, citations, and graph visualization.
2. FastAPI backend: upload orchestration, background indexing jobs, chat APIs, graph APIs, provider health checks, and persistence.
3. Microsoft GraphRAG workspace: file-based graph index outputs under `backend/ragtest/graphs/<graph_id>/`.

PostgreSQL is not used as the graph database. It stores application metadata and chat state. The actual GraphRAG index is stored in workspace files: Parquet tables and LanceDB vector-store outputs.

## GraphRAG Workspace Outputs

Each indexed graph is stored under:

```text
backend/ragtest/graphs/<graph_id>/
```

A typical workspace contains:

```text
input/
output/
cache/
settings.yaml
.env
```

Important GraphRAG outputs:

```text
documents.parquet
text_units.parquet
entities.parquet
relationships.parquet
communities.parquet
community_reports.parquet
output/lancedb/
```

The backend reads `entities.parquet` and `relationships.parquet` to build React Flow graph data. It also reads documents, text units, communities, and community reports for diagnostics and retrieval context. LanceDB is used by GraphRAG for embedding-backed retrieval.

The PoC deliberately keeps Microsoft GraphRAG's file, Parquet, and LanceDB output model instead of adding Neo4j. Neo4j would be useful for richer graph traversal APIs at larger scale, but it is not required for this proof-of-concept.

## Database Role

PostgreSQL stores application state:

- `indexed_graphs`: graph catalog, source filename, status, provider/model metadata, workspace path, active flag, counts, timestamps, and errors.
- `indexed_documents`: uploaded document metadata and final indexing counts.
- `conversations`: chat conversation metadata, optionally linked to a graph.
- `messages`: user/assistant messages, retrieved entities, retrieved relationships, retrieved sources, actual search mode, and routing reason.

The `indexed_graphs` table is central to the multi-graph design. A completed graph can be activated, which switches the active GraphRAG workspace used for graph view and chat queries.

## Quick Start With Docker Compose

Prerequisites:

- Docker and Docker Compose
- Free PostgreSQL port `5432`, or a custom `APP_PG_PORT`
- Ollama if using local models
- At least 8 GB RAM for smaller local models

1. Create the environment file:

```bash
cp .env.example .env
```

2. Ensure the GraphRAG workspace directories exist:

```bash
mkdir -p backend/ragtest/input backend/ragtest/output backend/ragtest/cache backend/ragtest/graphs
```

3. If using native Ollama, start it on the host:

```bash
ollama serve
ollama pull qwen2.5:3b
ollama pull nomic-embed-text
```

On Apple Silicon, native Ollama is recommended because it can use Metal acceleration. The default Docker Compose `APP_OLLAMA_BASE_URL` points to `http://host.docker.internal:11434`.

4. Start the services:

```bash
docker compose up -d --build
```

5. Follow backend logs:

```bash
docker compose logs -f backend
```

Production-mode URLs:

- App: http://localhost:8000
- OpenAPI: http://localhost:8000/docs
- ReDoc: http://localhost:8000/redoc
- Health check: http://localhost:8000/health
- Graph diagnostics: http://localhost:8000/api/graph/diagnostics

The backend creates database tables on startup. The Docker image initializes the GraphRAG workspace configuration before serving the app.

- GraphRAG indexing can issue many LLM and embedding calls.
- The backend uses conservative Gemini free-tier guards before indexing.
- The default Gemini embedding model is `gemini-embedding-2`.
- Existing `gemini-embedding-001` indexes are not compatible with `gemini-embedding-2`; change index and query embedding models together and reindex.

## Usage

1. Open the app: http://localhost:8000
2. Create or select a conversation.
3. Upload a `.txt` file with the `Upload Document` button.
4. Wait for indexing to finish.
5. Ask questions about the indexed document.
6. Open `Graph view` to inspect the knowledge graph.

Upload constraints:

- Only `.txt` files are accepted.
- Maximum upload size is 10 MB.
- Indexing can take a long time, especially with local models or large documents.

## Search Modes

- `Auto`: chooses a mode with lightweight query routing.
- `Local`: best for specific entities, direct facts, and local relationships.
- `Global`: best for whole-corpus summaries, themes, and patterns over community reports.
- `DRIFT`: useful for multi-hop, relationship-heavy, comparison, and influence questions.
- `Source`: application-specific mode for strict source extraction and direct source facts from raw text units.
- `Hybrid`: application-specific mode for source-grounded analytical synthesis using both raw source evidence and graph context.

Fallback behavior:

- `local`, `drift`, and `hybrid` require query embeddings.
- If `auto` selects an embedding-dependent mode and embeddings are unavailable, the backend falls back to `source`.
- If an automatically selected `global`, `drift`, or `hybrid` query fails or times out, the backend retries in `local`.
- Manual mode failures are returned as errors instead of silently changing the selected mode.

The assistant message stores the actual search mode and routing reason.

## Reindexing

Reindex documents when any of these change:

- embedding provider or embedding model,
- chunk size or chunk overlap,
- entity types,
- GraphRAG extraction settings,
- community report settings,
- cluster graph settings.

Changing only the query chat provider usually does not require reindexing.

Development reset:

```bash
docker compose down
docker compose up -d --build
```

Docker volume reset:

```bash
docker compose down -v
```

This removes PostgreSQL and Ollama Docker volumes. The host-mounted GraphRAG workspace under `backend/ragtest` can remain, so check `backend/ragtest/input`, `backend/ragtest/output`, and `backend/ragtest/graphs` when doing a full reindex.

## Observed Evaluation Results

Controlled corpus:

- Validates exact expected entities, direct relationship pairs, and short paths.
- Best suited for PoC regression checks.
- Passing result: `10` entities, `15` relationships, `9` expected relationship matches.

Einstein article tests:

- Small local models produced weak graph extraction and unreliable synthesis for broad questions.
- Gemini query models improved source discipline.
- Full Gemini indexing, Gemini embeddings, Gemini query generation, and `hybrid` retrieval produced the strongest source-grounded result.
- Retrieval still does not guarantee perfect synthesis; prompt constraints and evidence-use discipline remain necessary.

## Limitations

- Indexing can be expensive and slow because GraphRAG performs many LLM and embedding calls.
- Small local models such as `qwen2.5:3b` can produce weak graph extraction and unreliable synthesis.
- Query embedding provider/model must match the index embedding setup.
- `Local` can miss relevant source topics when the graph is noisy or the question is broad.
- `DRIFT` can be useful but may be slow and less directly auditable than source-bound answering.
- `Hybrid` improves grounding but still depends on the final model correctly using retrieved evidence.
- The frontend currently uploads one `.txt` file at a time.
- The graph is stored as GraphRAG files, not as a transactional graph database.

## Document Preparation Tips

- Use clean UTF-8 `.txt` files.
- Remove repeated headers, footers, and noisy boilerplate.
- Keep one file focused on one coherent topic or source unit.
- Split large corpora into smaller coherent files where possible.
- Re-upload sources after changing provider or embedding model settings.
