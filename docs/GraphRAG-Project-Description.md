# GraphRAG Project Description

## 1. Project Context

This repository is a proof-of-concept GraphRAG application for a university research topic about knowledge-graph-enhanced generative systems. The project investigates how retrieval-augmented generation can be improved when the system does not rely only on vector similarity over text chunks, but also builds an explicit entity and relationship structure from the source document.

The practical goal is source-grounded question answering over uploaded `.txt` documents. The application indexes the document with Microsoft GraphRAG, stores the generated graph artifacts, visualizes extracted entities and relationships, and lets the user ask questions through multiple retrieval modes. The research motivation is explainability and hallucination reduction: a useful answer should be connected to concrete source text, visible entities, visible relationships, and query metadata that can be inspected after generation.

Pure vector RAG is useful for retrieving semantically similar passages, but it can struggle with whole-corpus questions, multi-hop relationships, and questions where the relevant evidence is distributed across several parts of the document. Microsoft GraphRAG addresses this by extracting entities, relationships, communities, summaries, and embeddings from unstructured text. Its indexing documentation describes a configurable pipeline that extracts structured data, detects communities, generates community reports, embeds text, and stores outputs as Parquet tables plus vector-store embeddings. This project uses that pipeline as the core indexing engine and wraps it in a full-stack application.

## 3. High-Level Architecture

The application has three main layers:

1. Frontend user interface: upload, provider selection, graph selection, chat, citations, and graph visualization.
2. FastAPI backend: upload orchestration, background indexing jobs, chat APIs, graph APIs, provider health checks, and database persistence.
3. Microsoft GraphRAG workspace: isolated file-based index outputs under `backend/ragtest/graphs/<graph_id>/`.

The important architectural distinction is that PostgreSQL is not the graph database. PostgreSQL stores application metadata and chat state. Microsoft GraphRAG stores the actual graph index as files in each workspace.

## 4. Backend Indexing Flow

The document upload flow starts at:

```text
POST /api/index/upload
```

The endpoint accepts one `.txt` file and optional provider selections. It creates:

- a new `document_id`,
- a new `graph_id`,
- an `indexed_graphs` row,
- an `indexed_documents` row,
- an in-memory `IndexJob` progress object.

The backend then schedules `run_indexing_job` as a FastAPI background task. Each graph receives an isolated workspace path:

```text
backend/ragtest/graphs/<graph_id>/
```

Inside the background job, a new `GraphRAGService` instance is created, assigned to the graph workspace, and used to verify the selected chat and embedding providers. The actual indexing call is:

```python
stats = await worker.index_document(
    filename,
    content,
    progress_callback=update_progress,
    index_chat_provider=index_chat_provider,
    index_embed_provider=index_embed_provider,
)
```

`GraphRAGService.index_document` prepares the workspace, writes the uploaded file to the workspace `input` directory, loads the GraphRAG configuration, and then calls the Microsoft GraphRAG Python API:

```python
index_result = await asyncio.wait_for(
    api.build_index(config=self.config),
    timeout=settings.graphrag_index_timeout_seconds,
)
```

This matches the official GraphRAG API overview, where a `GraphRagConfig` is loaded from `settings.yaml` and `api.build_index(config=...)` is used to build the index. The application monitors GraphRAG log progress to report chunk extraction and summarization status back to the frontend.

After GraphRAG finishes, the service invalidates cached data and counts the generated entities and relationships. The database rows are updated to `completed` or `failed`, including the final entity and relationship counts.

## 5. GraphRAG Workspace And File Outputs

Each indexed graph is stored under:

```text
backend/ragtest/graphs/<graph_id>/
```

A typical workspace contains:

```text
input/
output/
cache/
settings.yaml
.env
```

The important index outputs are in `output/`:

```text
documents.parquet
text_units.parquet
entities.parquet
relationships.parquet
communities.parquet
community_reports.parquet
output/lancedb/
```

The backend reads `entities.parquet` and `relationships.parquet` to build the React Flow graph. It also reads documents, text units, communities, and community reports for diagnostics and query context. LanceDB is used by Microsoft GraphRAG for embedding-backed retrieval, especially entity description and community report vector searches.

The current PoC deliberately uses file, Parquet, and LanceDB storage instead of Neo4j. That keeps the project simpler and aligned with Microsoft GraphRAG's default outputs. A graph database such as Neo4j would be useful for richer graph querying, persistent graph traversal APIs, or multi-document graph management at larger scale, but it would add operational complexity that is not needed for this proof-of-concept.

## 6. Database Design

PostgreSQL stores application state, not the full graph index.

The main tables are:

- `indexed_graphs`: graph catalog, source filename, status, provider metadata, model metadata, workspace path, active graph flag, counts, timestamps, and error message.
- `indexed_documents`: uploaded document metadata and final indexing counts.
- `conversations`: chat conversation metadata, optionally linked to a graph.
- `messages`: user and assistant messages, retrieved entities, retrieved relationships, retrieved sources, actual search mode, and routing reason.

The `indexed_graphs` table is central to the multi-graph design. A completed graph can be activated, which updates the database and switches the active `GraphRAGService` workspace. Conversations can also be associated with a graph so chat history remains tied to the source index used for the answer.

## 7. Query Modes

The chat endpoint is:

```text
POST /api/chat/conversations/{conversation_id}/query
```

The frontend sends:

- the user question,
- the selected search mode,
- query chat provider,
- query embedding provider.

The backend supports these modes:

- `auto`
- `local`
- `global`
- `drift`
- `source`
- `hybrid`

### Auto

`auto` is a lightweight router in `GraphRAGService.route_search_mode`. It checks query wording and chooses a mode:

- source-bound extraction requests route to `source`,
- source-bound analytical synthesis routes to `hybrid`,
- overview and dataset-level prompts route to `global`,
- multi-hop relationship prompts route to `drift`,
- specific entity/detail prompts route to `local`,
- source-grounded GraphRAG PoC pipeline questions route to `source` or `hybrid`, depending on whether the query asks for a direct fact or analytical synthesis.

The resolved mode and explanation are saved with the assistant message.

### Local

`Local` uses Microsoft GraphRAG `LocalSearch` with `LocalSearchMixedContext`. It is best for specific entities, direct facts, and local relationships. Microsoft's documentation describes Local Search as combining knowledge-graph structured data with raw document chunks for questions about specific entities.

In this project, Local Search loads entity, relationship, community report, and text-unit objects, connects to the LanceDB entity-description vector store, and uses a custom prompt that requires citations to exact entities and relationship pairs.

### Global

`Global` uses Microsoft GraphRAG `GlobalSearch` over community reports. It is intended for whole-document or dataset-level questions such as summaries, themes, and broad patterns. The GraphRAG documentation describes Global Search as a map-reduce style process over generated community reports.

The project sets `allow_general_knowledge=False` so the answer should stay within indexed data.

### DRIFT

`DRIFT` uses Microsoft GraphRAG `DRIFTSearch`. It combines community-level context and local exploration, making it suitable for multi-hop or relationship-heavy questions. Microsoft's DRIFT documentation describes it as combining global and local search characteristics to broaden the query starting point and refine with follow-up exploration.

In the PoC, DRIFT is useful for complex relationship questions, but earlier tests showed that it can be slow and less directly auditable if the answer relies on generated community context instead of raw source excerpts.

### Source

`Source` is an application-specific mode. It ranks raw `text_units.parquet` evidence and answers from explicit source text. It is used for strict source extraction and direct source facts. It returns `retrieved_sources` with `[S1]`, `[S2]` style citation metadata, including source filename, text-unit id, excerpt, and document id where available.

The GraphRAG PoC sample test shows why this mode matters: direct source-grounded pipeline questions can be answered from raw text units without forcing an expensive DRIFT query.

### Hybrid

`Hybrid` is also application-specific. It combines ranked raw source evidence with graph context from Local-style entity and relationship retrieval. Its prompt states that source excerpts are authoritative and graph context can help organize relationships but cannot override sources.

This is the strongest mode for source-grounded analytical synthesis in the current PoC because it gives the model both source excerpts and graph structure.

### Fallback Behavior

Some modes require query embeddings. The backend checks the configured query embedding provider before running `local`, `drift`, or `hybrid`. If `auto` selects an embedding-dependent mode and embeddings are unavailable, the system falls back to `source`. If an automatically selected `global`, `drift`, or `hybrid` query fails or times out, the backend retries in `local`. Manual mode failures are surfaced as errors instead of silently changing the user's selected mode.

## 8. Frontend Behavior

The main frontend screen is a working application, not a landing page. The user can:

- select index chat and index embedding providers,
- select query chat and query embedding providers,
- upload a `.txt` document,
- monitor indexing progress,
- view the active indexed graph,
- switch between indexed graphs,
- create and select conversations,
- choose a search mode for each message,
- inspect assistant answers, source citations, entity citations, and relationship citations,
- open graph visualization,
- click entity and relationship citations to highlight graph nodes.

The graph visualization uses React Flow. Backend graph data is transformed from GraphRAG entity and relationship Parquet rows into nodes and edges. The UI supports graph view modes such as main, connected, and all, and it highlights selected nodes or relationship endpoints from chat citations.

## 9. Provider Configuration And Run Modes

The application separates indexing and querying providers:

```env
APP_INDEX_CHAT_PROVIDER=ollama|gemini|openrouter
APP_INDEX_EMBED_PROVIDER=ollama|gemini|openrouter
APP_QUERY_CHAT_PROVIDER=ollama|gemini|openrouter
APP_QUERY_EMBED_PROVIDER=ollama|gemini|openrouter
```

The default local setup uses Ollama:

```env
APP_OLLAMA_BASE_URL=http://host.docker.internal:11434
APP_OLLAMA_LLM_MODEL=qwen2.5:3b
APP_OLLAMA_EMBED_MODEL=nomic-embed-text
```

Gemini is optional and currently configured around:

```env
APP_GEMINI_LLM_MODEL=gemini-3.1-flash-lite
APP_GEMINI_EMBED_MODEL=gemini-embedding-2
```

OpenRouter is optional for chat, with embeddings only possible if the selected OpenRouter model supports embeddings through `/embeddings`. The README recommends keeping query embedding provider/model aligned with the index embedding provider/model. If they differ, LanceDB retrieval can become inaccurate and `/api/graph/diagnostics` emits a warning.

## 10. Tests And Observed Results

### GraphRAG PoC Sample

The GraphRAG PoC test corpus is in:

```text
backend/samples/graphrag_poc/
```

It contains a small deterministic course-documentation corpus with expected entities and relationships. The content directly describes the project's GraphRAG pipeline: ingestion, chunking, embeddings, vector storage, graph storage, retrieval, answer generation, citations, and evaluation.

Because the upload UI accepts one file at a time, the three source files were merged into:

```text
graphrag_poc_all.txt
```

Technical note: the original three sample documents are still kept as separate `.txt` files in the sample directory. The merged `graphrag_poc_all.txt` file is used to validate the current one-file upload flow, and its `Source:` labels preserve the boundaries of the original three source documents.

Expected entities include:

- `GraphRAG PoC`
- `Document Loader`
- `Chunking Module`
- `Indexing Pipeline`
- `Embedding Model`
- `Vector Store`
- `Entity Extractor`
- `Relationship Extractor`
- `Graph Store`
- `Query Engine`
- `Answer Generator`
- `Evaluation Script`
- `graph traversal`
- `text chunks`

- `vector retrieval`

The frontend visualization showed two connected components in the same indexed graph. The main component contains the operational pipeline around `Graph Store`, `Document Loader`, `Chunking Module`, `Vector Store`, `Query Engine`, and `Answer Generator`. A smaller separate component contains the high-level retrieval-strategy structure:

```text

```

This separation is expected. GraphRAG extracted the implementation pipeline and the high-level retrieval-strategy statement, but it did not create an additional bridge edge from `GraphRAG PoC` to the main implementation component. The graph is still valid because the validator checks both components.

Detailed report:

```text
docs/GraphRAG-PoC-Sample-Test-Report.md
```

### First Test

The first test compared plain `qwen2.5:3b` with GraphRAG using `qwen2.5:3b` on an Albert Einstein article. GraphRAG recognized the requested analytical structure better, but factual reliability was poor. It produced fabricated dates, geographic errors, weak source faithfulness, and citation-like text that was not trustworthy.

Conclusion: the small local model was not reliable enough for broad source-grounded synthesis.

### Second Test

The second test used `qwen2.5:3b` for indexing, Gemini for querying, and forced `Local` mode with split questions. Local mode improved focused facts, especially the Nobel Prize question, but retrieval missed broad source topics such as the 1905 annus mirabilis papers, the 1933 political transition, and the Roosevelt letter.

Conclusion: a better query model helped, but Local retrieval over a noisy index was not enough for broad historical synthesis.

### Third Test

The third test compared baseline Gemini 3.1 Flash Lite with a GraphRAG-indexed Gemini setup. GraphRAG improved source grounding and auditability, using source references and a more controlled structure. The baseline answer was smoother, but less auditable and more likely to include unsupported outside knowledge.

Conclusion: GraphRAG was better aligned with the project's source-grounded objective, despite remaining language and prompt-compliance issues.

### Fourth Test

The fourth test used full Gemini indexing, Gemini embeddings, Gemini query generation, and `Hybrid` retrieval. This produced the strongest source-grounded result so far. It combined raw source excerpts with graph context, returned `[S]` citations, and reduced unsupported claims compared with the baseline.

The rerun with `gemini-embedding-2` kept strong source retrieval but showed that final synthesis can still regress. In particular, one run misclassified the 1905 annus mirabilis achievements by replacing mass-energy equivalence with the PhD dissertation.

Conclusion: Hybrid mode with strong indexing/query providers best demonstrates the PoC value, but retrieval does not guarantee perfect synthesis. Prompt constraints and evidence-use discipline remain necessary.

## 11. Graph Quality Comparison

The GraphRAG PoC sample and Einstein article demonstrate two different graph-quality regimes.

The GraphRAG PoC sample is small, deterministic, and aligned with the project documentation. It has a limited set of expected pipeline entities and relationships, so validation can check exact entities, direct edges, and short paths. It is appropriate for regression tests because it validates the same concepts the project is meant to demonstrate.

The Einstein article is broad, biographical, scientific, and political. It produces many more entities and relationships, including noisy or overloaded relationships. Earlier tests found combined multi-person entities, low-relevance graph nodes, broad relationship descriptions, and retrieval gaps for important topics. This is a better stress test for source grounding and synthesis, but it is harder to evaluate mechanically.

Graph quality depends on:

- indexing chat model quality,
- embedding model and consistency between index/query embeddings,
- chunk size and overlap,
- query mode,
- prompt constraints,
- whether the answer uses raw source excerpts or only graph/community context.

## 12. Technology Choices

### Microsoft GraphRAG Instead Of Pure Vector RAG

Microsoft GraphRAG is appropriate because the research topic is specifically about knowledge-graph-enhanced generation. It extracts entities, relationships, communities, and summaries rather than only embedding chunks. This supports explainability, graph visualization, and multi-hop relationship reasoning.

### Parquet And LanceDB Instead Of Neo4j

The PoC keeps Microsoft GraphRAG's default file-based outputs. `entities.parquet`, `relationships.parquet`, `text_units.parquet`, `communities.parquet`, `community_reports.parquet`, and LanceDB outputs are enough to inspect, query, and visualize the generated index. Neo4j remains a possible future extension, but it is not required for the current proof-of-concept.

### PostgreSQL For Metadata

PostgreSQL is used for application metadata because it is reliable for catalog rows, chat history, message JSON fields, active graph state, and provider metadata. It is not used as a graph traversal database.

### FastAPI And React Flow

FastAPI is a good fit for background indexing jobs, typed API schemas, provider checks, and serving both APIs and the production frontend. React Flow is a good fit for exploring graph nodes and edges because the backend already emits node/edge structures derived from GraphRAG outputs.

## 13. Limitations

- Indexing can be expensive and slow because Microsoft GraphRAG performs many LLM and embedding calls.
- Small local models such as `qwen2.5:3b` can produce weak graph extraction and unreliable synthesis.
- Query embedding provider/model must match the index embedding setup for reliable LanceDB retrieval.
- `Local` can miss relevant source topics when the graph is noisy or the question is broad.
- `DRIFT` can be useful but may be slow and less directly auditable than source-bound answering.
- `Hybrid` improves grounding but still depends on the final model correctly using the retrieved evidence.
- The frontend currently uploads one `.txt` file at a time.
- The graph is stored as GraphRAG files, not as a transactional graph database.
